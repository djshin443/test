## 2. 새로 발견된 문제 (5건)

### 2.1 사이드바 ↔ INDEX 카테고리 불일치

| 위치 | 카테고리 구분 |
|------|-------------|
| **사이드바** | 4개: 설계 문서, 운영 가이드, **교육 자료**(13개 전부), 변경 이력 |
| **INDEX 본문** | 5개: 설계 문서, 운영 가이드, **교육 자료**(4개), **인터랙티브**(9개), 변경 이력 |

**문제**: INDEX에서는 "교육 자료"와 "인터랙티브"가 분리되어 있는데, 사이드바에서는 "교육 자료" 하나에 13개가 전부 들어있음. 동일한 분류 체계를 양쪽에 적용해야 사용자가 혼란하지 않음.

**수정 방안**: 사이드바에서도 "교육 자료"와 "인터랙티브"를 별도 `<details>` 그룹으로 분리

### 2.2 사이드바 영문 이름 잔존 (2건)

| 사이드바 표시 | 수정 제안 |
|-------------|-----------|
| "OPERATION TOOLS" | "운영 도구 사용 가이드" (INDEX 본문과 일치시킴) |
| "ROADMAP" | "개발 로드맵" (INDEX 본문과 일치시킴) |

### 2.3 SYSTEM_LOGIC.md 내부 불일치

SYSTEM_LOGIC.md의 주요 특징 란:
- **개요**: "Suricata 전용 **30차원** 특징"
- **feature_extractor.py 설명**: "**30차원** Suricata 전용 특징 추출"

하지만 같은 문서 내의:
- **4.1 Autoencoder**: "입력(**30**차원)" — 실제로는 59차원이 StandardScaler를 거쳐 입력됨
- MODULE_TREE.md의 feature_extractor.py 설명: "**59개** 피처 (30개 표준 + 3개 확장)"

**수정 방안**: SYSTEM_LOGIC.md 개요를 "59차원 특징 (30기본+3확장+26페이로드)" 또는 최소 "기본 30차원 + 확장 29차원 = 59차원"으로 명확하게 표기

### 2.4 MODULE_TREE.md의 문서 트리와 실제 사이드바 불일치

MODULE_TREE.md에 기재되어 있지만 사이드바에 없는 문서:

| 문서 | MODULE_TREE 내 기재 | 사이드바 |
|------|-------------------|---------|
| `docs/reference/API_REFERENCE.md` | "API 레퍼런스 (사이드바 숨김)" | 없음 |
| `docs/guides/README.md` | "빠른 시작" | 없음 |
| `docs/reference/ARCHITECTURE.md` | "아키텍처 설계" | 없음 |

"사이드바 숨김"이라 의도적일 수 있으나, README.md(빠른 시작)는 사이드바에 노출하는 것을 권장.

### 2.5 PRESENTATION_EASY.md 오탈자

- 1절: "스스**로** 똑똑해지는" → 정확하지만 시스템 전체에 "스스로"가 반복 사용됨

## 3. 로드맵 리뷰 (Phase 3~5)

### 현재 로드맵 평가

로드맵이 크게 개선되었습니다. Phase 3에 벡터DB+LLM을 배치한 것이 특히 좋습니다.

### Phase 3 (벡터DB + LLM) — 평가: 매우 좋음

| 항목 | 평가 |
|------|------|
| ChromaDB 벡터DB | 적절. 경량이고 파이썬 네이티브. ~7MB/월 추정도 합리적 |
| Ollama + Qwen 14B | GPU 메모리 배분(260MB 추론 + 10GB LLM + 14GB 여유) 계산이 현실적 |
| 관리자 챗봇 (RAG) | "이 IP 분석해줘" 시나리오는 실용적. 시연 임팩트 높음 |
| 일일 보안 리포트 + 이메일 | 운영 자동화의 핵심. Phase 3에 맞는 위치 |
| Unknown 자동 분류 | HDBSCAN → AI API → 관리자 승인 플로우가 체계적 |

**추가 제안**: Phase 3에 **Slack/Teams 웹훅 알림**을 포함시키면 좋겠음. 이메일 리포트는 일일 요약에 적합하지만, CRITICAL 탐지 시 즉시 알림은 메신저가 더 효과적.

### Phase 4 (ClickHouse + Grafana) — 평가: 좋음

| 항목 | 평가 |
|------|------|
| ClickHouse + Grafana | 시계열 데이터에 적합. 탐지 메트릭 시각화에 최적 |
| 차단 IP 영구 DB | 메모리 FIFO → DB 전환은 필수적 개선 |
| 모델 드리프트 모니터링 | FPR/탐지율 시계열 추적은 MLOps 핵심 |
| 위협 인텔리전스 연동 | VirusTotal/AbuseIPDB 연동으로 IP 평판 자동화 |

**추가 제안**:
- **GeoIP 시각화**: Grafana 대시보드에 세계 지도 패널(Geomap) 추가하면 시각적 효과 극대화
- **SIEM 연동**: ClickHouse 사용 시 Grafana → Elasticsearch로의 연동도 자연스러움. 기업 고객 도입 시 거의 필수 요구사항

### Phase 5 (확장 + 자동화) — 평가: 방향 좋음, 보완 필요

| 항목 | 평가 |
|------|------|
| LLM 서버 분리 | 탐지(CPU) + LLM(GPU) 분리는 스케일 아웃에 필수 |
| Kafka + Flink 분산 처리 | 10Gbps+ 대응. 다만 현실적으로 Phase 6급 난이도 |
| Red Team MITRE ATT&CK | 자동 시뮬레이션 → 미탐 수집 → 재학습은 이상적이나 구현 난이도 매우 높음 |

**추가 제안 (Phase 5 또는 신규 Phase 6)**:

| 항목 | 설명 | 이유 |
|------|------|------|
| Docker/K8s 배포 | tar.gz → 컨테이너 원클릭 배포 | 기업 고객 PoC 시 배포 속도 결정적 |
| 컴플라이언스 리포트 | ISMS-P / ISO 27001 감사 자동 보고서 | 국내 기업 도입 시 인증 필수 |
| 암호화 트래픽 분석 | JA3/JA4 핑거프린팅 | TLS 비중 80%+ 시대에 대응 필수 |
| MITRE ATT&CK 매핑 | 11개 공격→ATT&CK Tactic/Technique | 보안팀과의 공통 언어 |

## 4. 잘 된 점 (1차 대비 추가)

### 4.1 운영 도구 체계화

수정 후 운영 도구가 매우 체계적으로 정리됨:

```
./run.sh           → 탐지 시작 (GPU 추론서버 자동)
./stop.sh          → 탐지 중지
./check_queue.sh   → 시스템 상태 확인
./check_model.sh   → 모델 검증 (재학습 후 필수)
./train_gpu.sh     → 모델 학습 (메뉴형)

scripts/
├── backup_models.sh     → backup/list/restore/delete/diff
├── export_system.sh     → deploy/full/code/restore/list
└── retrain_model.sh     → xgb/ae/if/scaler/all/status
```
